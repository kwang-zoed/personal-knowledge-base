### Function Calling是如何实现的

> **（原图转文字）庖丁解牛：Function Calling 三大底层实现原理**
>
> 1. **协议与微调（SFT）**：模型并非天生会写 JSON；在监督微调阶段注入大量 `[Tools] + [Query] = [Call]` 语料，教会模型识别「何时该调用」的触发特征。
> 2. **特殊标识符（Tokens）**：使用内部 Special Tokens 隔离边界（如 `<tool_call>`）；预测到标签后触发底层推理引擎切换状态，进入结构化生成模式。
> 3. **约束解码（Constrained）**：解决漏括号、引号等问题；推理时用 FSM 或 GBNF，在采样层直接过滤不符合 JSON Schema 的非法 Token。


##### 约束解码详解

**FSM**有限状态机做校验：把 JSON / 函数调用的语法，转换成一个有限状态机（FSM）

比如：`{` → `"key":` → `value` → `,` → 下一个键值对 → `}`

解码过程中，FSM 会跟踪当前状态，比如 “刚写完 `{`，接下来必须是 `"`”

模型预测出的下一个 Token，如果会让状态机进入 “非法状态”，就直接被屏蔽掉，不参与采样

**GBNF** 是一种扩展的巴科斯范式，用来描述 JSON 这样的结构化语言的语法

比如定义：`json = object / array; object = "{" (pair ("," pair)*)? "}";`

解码器会根据这个语法，在每一步计算 “当前位置能生成哪些合法 Token”，然后只让模型在这个集合里选

相当于给模型一个 “合法词表白名单”，超出的 Token 根本没机会被选中

#### FunctionCalling生产中会遇到的一些问题

> **（原图转文字）避坑指南：生产环境难点与对策**
>
> | 难点 | 对策 |
> | --- | --- |
> | **参数幻觉**：凭空捏造 API 不存在的参数，或不必要地强制调工具 | **Few-shot 示例 + 动态检索**：正负调用样例写入 Prompt；RAG 按 Query 只注入最相关 3–5 个工具 |
> | **格式不稳定阻断流程**：JSON 语法错误导致反序列化崩溃 | **强类型校验 + 重试**：开启 Strict Mode；后端用 Pydantic 校验，异常信息回灌模型自修正 |
> | **安全越权风险**：Prompt Injection 诱导调用删库、转账等高危 API | **中间层沙盒 + HITL**：代码层硬鉴权；敏感操作弹窗人工确认 |


> **（原图转文字）Function Calling 三点收束**
>
> 1. **认清本质**：Function Calling 不是模型去执行代码，而是受限的、结构化的参数文本生成。
> 2. **点出核心**：底层靠 SFT 训练识别能力，配合推理层 FSM 约束解码，保证 JSON 准确率。
> 3. **落地视角**：生产重点不在模型多聪明，而在后端错误重试闭环与高危拦截机制。
