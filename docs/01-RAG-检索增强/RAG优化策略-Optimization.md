## RAG优化策略

### 1. Simple RAG(固定切分)

硬性规定chunk大小，不顾及chunk间的关联

### 2. Semantic Chunking(相邻句子相似度分块)

不在规定chunk大小，而是计算相邻句子间的相似度，相似度低于阈值进行分块

具体实现

```python
# 不依赖langchain的方法
import numpy as np
import nltk
from sentence_transformers import SentenceTransformer

nltk.download('punkt', quiet=True)
from nltk.tokenize import sent_tokenize

# 1. 加载模型（轻量高效）
model=SentenceTransformer('all-MiniLM-L6-v2')  # 英文；中文用 'all-MiniLM-L6-v2-zh'

def semantic_chunking(text, threshold=0.7, buffer_size=1):
    # 2. 分句
    sentences=sent_tokenize(text)
    if len(sentences) <= 1:
        return [text]
    
    # 3. 加上下文窗口（提升向量质量）
    combined_sentences = []
    for i in range(len(sentences)):
        start=max(0, i-buffer_size)
        end=min(len(sentences), i+1+buffer_size)
        combined = " ".join(sentences[start:end])
        combined_sentences.append(combined)
    
    # 4. 生成向量
    embeddings=model.encode(combined_sentences, convert_to_numpy=True)
    
    # 5. 计算相邻相似度
    similarities = []
    for i in range(len(embeddings)-1):
        sim=np.dot(embeddings[i], embeddings[i+1])/(np.linalg.norm(embeddings[i])*np.linalg.norm(embeddings[i+1]))
        similarities.append(sim)
    
    # 6. 按阈值切分
    chunks = []
    current_chunk = [sentences[0]]
    for i in range(len(similarities)):
        if similarities[i] < threshold:
            chunks.append(" ".join(current_chunk))
            current_chunk = [sentences[i+1]]
        else:
            current_chunk.append(sentences[i+1])
    chunks.append(" ".join(current_chunk))  # 最后一块
    
    return chunks



# ---------------------------------------------------------------------------------------
# langchain调库方法
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

# 1. 初始化 Embedding（OpenAI 或本地模型）
embeddings=OpenAIEmbeddings()  # 需 OPENAI_API_KEY
# 本地替代：from langchain.embeddings import HuggingFaceEmbeddings; embeddings=HuggingFaceEmbeddings(model_name='all-MiniLM-L6-v2')

# 2. 语义分块器（推荐 percentile 方式）
splitter=SemanticChunker(
    embeddings=embeddings,
    breakpoint_threshold_type="percentile",  # 三种：percentile/standard_deviation/interquartile
    breakpoint_threshold_amount=90,  # 越高块越大（90=只切最不相关的10%）
    buffer_size=1  # 上下文窗口
)

# 3. 直接切分
text = """你的长文本..."""
chunks=splitter.split_text(text)
```

### 3.small-to-big Retrieval(父子文档关联检索)

**第一步：全局文档 → 切分【父块】**

按章节 / 大段落切，保证**语义完整、上下文不割裂**。

**第二步：父块全部存入【外部存储】**

用字典 / Redis / 数据库，建立映射：

```
parent_id → 完整父块原文
```

**第三步：每个父块内部 → 再切【子块】**

子块粒度很小，适合向量匹配、检索精准。

**第四步：子块入库向量库**

向量库只存：

- 子块向量
- 子块文本
- **关联的 parent_id**（只存 ID，不存父块全文）

**第五步：用户提问 → 向量库检索 Top-K 子块**

召回最相关的细粒度子块。

**第六步：从子块元数据取出 parent_id**

去重，拿到一批唯一父块 ID。

**第七步：用 parent_id 从外部存储取出完整父块**

拼成完整上下文。

**第八步：把拼接好的父块上下文 + 问题塞进 Prompt 给 LLM**

#### 检索到子块 → 拿整段父块全灌进 Prompt==>token消耗过大改进

#### 方案 1：窗口扩展（最常用）

不拿**整个父块**，只拿**命中子块 + 前后若干句 / 前后若干字符**

- 检索命中细粒度子块

- 在父块里，只截取该子块**上下文窗口**

- 例如：子块前后各扩展 150 token

  既保留完整语义，又不塞整段父块，大幅省 Token

> 本质：小检索 → 局部放大，不是全局放大

#### 方案 2：父块先摘要，再进 Prompt

1. 父块存在外部存储
2. 提前给每个父块生成**精简摘要**
3. 检索到 parent_id 后：优先塞摘要 + 关键片段，不塞全文Token 直接砍一半以上。

#### 方案 3：Rerank 再过滤（工程标配）

流程改成：

1. 向量检索 Top-K 子块 → 拿到一批 parent_id
2. 取出对应父块全文
3. 用 Rerank 模型**对父块片段做相关性打分**
4. 只把高分片段塞进 Prompt,低价值冗余直接扔掉，控制 Token 总量。

# 2. 初始化Embedding模型

### 6.Document Augmentation

在索引阶段，让LLM针对每个chunk预生成几个可能的问题/摘要/关键词，检索时匹配这些预设问题。


### 7.Query Transformation

包括查询重写，回溯提示，子查询分解

> **（原图转文字）Query Transformation 三种重写方式对照**
>
> | 重写方式 | 系统提示词要点 | 用户提示词要点 |
> | --- | --- | --- |
> | **查询重写（Query Rewriting）** | 把查询改写得更具体、更利于检索 | 补充相关术语与概念，提高召回准确度 |
> | **回溯提示（Step-back Prompting）** | 生成更宽泛的上位问题以取背景知识 | 从具体问题抽象出更一般的检索问题 |
> | **子查询分解（Sub-query Decomposition）** | 把复杂问题拆成若干简单子问题 | 各子问题覆盖不同侧面，合起来回答原问 |




### 8.Reranker

通过向量匹配初筛出Topk×n，再用高精度二分排位模型精选Topk



### 9.RSE(Relevcant Segment Extration)

相似度过滤-->上下文窗口查找-->片段总分值计算-->阈值筛选

> **（原图转文字）RSE 算法流程示例**
>
> 1. **相似度过滤后（向量得分）**：`[0.8][0.7][0.2][0.1][0.6][0.5][0.1][0.2]`，其中 0.8/0.7/0.6/0.5 为高相关。
> 2. **上下文窗口查找（窗口长度=2）**：滑窗形成片段，如片段1=`[0.8][0.7]`，片段2=`[0.6][0.5]`，以及跨边界的 `[0.7][0.2]`、`[0.1][0.6]`、`[0.5][0.1]` 等。
> 3. **片段总分值**：片段1 = 0.8+0.7=1.5；片段2 = 0.6+0.5=1.1；片段3 = 0.7+0.2=0.9；再按总分与阈值筛选。


### 10.Contextual Compression

> **（原图转文字）Contextual Compression**
>
> **定义**：多取一些 chunk，再压缩，只保留与 query 相关的句子再交给 LLM。  
> **假设**：检索结果常含无关信息，需要压缩提炼。  
> **实现**：用提示词让模型对长片段「脱水」，只留干货。  
> **优势**：减少 Token 浪费；减少背景干扰。
>
> **典型 system prompt 要点**：只提取与查询直接相关的句子/段落；保持原文措辞与顺序；可保留略冗余相关内容；排除无关文本；纯文本输出、不加注释。




### 11.Feedback loop

> **（原图转文字）Feedback Loop（反馈闭环）— 概念与示例**
>
> **定义**：记录用户交互与 LLM 反思，动态调整文档权重，实现检索自进化。  
> **假设**：系统可从用户反馈持续改进检索质量。  
> **流程**：用户查询 → 检索 → 生成 → 收集反馈 → 存储并反思 → 权重重标定。  
> **优势**：系统自进化；用户偏好驱动；持续提升准确率。
>
> **示例**：问「什么是神经网络？」召回文档 A/B/C（相似度 0.8/0.6/0.5），基于 A 生成回答；用户给相关性 4、质量 5，并评论「清晰但可加技术细节」。


> **（原图转文字）Feedback Loop — 存储与后续优化**
>
> **反馈 JSON 示例字段**：query / response / relevance / quality / comments / timestamp。  
> **备注**：可用大模型判断哪些 chunk 对当前问题更相关，再调权重。
>
> **后续查询**：如「神经网络的基本原理是什么？」——有正反馈的文档 A 分数可上调（例：0.8 × 1.2 = 0.96），无反馈文档保持原分。
>
> **数据流**：用户查询 → 检索文档 → 生成回答 → 收集反馈 → 存储反馈 → 优化后续检索。






### 12.Self RAG

> **（原图转文字）Self-RAG：模型自我反思**
>
> **定义**：是否需要检索？检索结果是否相关？生成是否有证据支撑？  
> **假设**：模型可过滤相关检索结果、降噪，用高相关片段作答。  
> **逻辑**：先判要不要检索 → 否：直接答；是：向量库取 top-k → 相关性评估只留相关片段 → 基于上下文生成。  
> **优势**：极其严谨；幻觉率极低。
>
> **流程**：开始 → 判断是否检索外部文档 →（否）直接生成 /（是）文档检索 → 相关性评估 → 答案生成。




### 13.Knowledge Graph

> **（原图转文字）Knowledge Graph / GraphRAG**
>
> **定义**：构建实体与关系图谱，跨节点检索关联信息。  
> **假设**：信息存在关联，图结构更能表达关系。  
> **逻辑**：从文档抽 Entity/Relationship；检索时除向量找实体外，沿关系边做图遍历，挖跨章节/跨文档隐性关联。  
> **优势**：长链推理；挖掘隐性关联。
>
> **示意**：Artificial Intelligence → Machine Learning → Deep Learning / NLP → CNN、RNN、Attention → Transformers → RAG。






### 14.Hierarchical Indices

> **（原图转文字）Hierarchical Indices（层次索引）**
>
> **定义**：先检索大块 Summary 锁定范围，再在对应 Summary 下检索精细 Chunk。  
> **假设**：层次索引可平衡精确度与上下文。  
> **逻辑**：两级向量库（Summary / Chunk）；Query 先在 Summary 粗筛 TopK，再在相关章节的 Chunk 库精筛。  
> **优势**：缩短检索路径；平衡全局与局部。适合法律文书、科研论文等。
>
> **索引**：原始文档 → Summary 粗分块 + Chunk 细分块 → 分别向量化入库。  
> **查询**：Query → Summary TopK → 对应 Chunk TopK → 拼接上下文 → LLM 出最终答案。


一、前置准备：建立关联关系（索引阶段）
这一步是 “把大块摘要和它的小块内容绑定”，为后续检索铺路：

​		文档分层处理
将原始文档按章节 / 段落 / 逻辑单元切分为大块（Summary Block），为每一块生成一个文本摘要（Summary），并给每个大块分配唯一标识（如block_id、section_id）。
再将每个大块进一步切分为更细粒度的Chunk，每个 Chunk 都继承所属大块的标识（如在 Chunk 的元数据中添加block_id: xxx字段）。

​		双向量库构建
Summary 向量库：存储每个大块的摘要文本、向量，以及对应的block_id。
Chunk 向量库：存储所有细粒度 Chunk 的文本、向量，以及所属的block_id元数据。
此时，block_id就成了 Summary 和 Chunk 之间的 “桥梁”，所有 Chunk 都知道自己属于哪个 Summary 大块。

二、检索阶段：从 Summary 到 Chunk 的关联步骤
用户提问后，按 “先粗后细” 的流程完成关联：
粗检索：锁定相关 Summary
将用户 Query 向量化，在Summary 向量库中进行相似度检索，召回 Top-K 个最相关的摘要（如召回 3 个最匹配的大块）。
从召回的 Summary 中，提取它们对应的block_id列表（比如[block_12, block_45, block_78]）。
细检索：在对应 Chunk 库中过滤召回
将用户 Query 再次向量化，在Chunk 向量库中进行检索，但增加元数据过滤条件：只检索元数据中block_id属于上一步列表的 Chunk。



### 15.HDE(Hypothetical Document Embeddings)

> **（原图转文字）HyDE 核心定义**
>
> 先让 LLM 生成一个「假答案」，用假答案的向量去检索真文档。  
> **假设**：假设性答案文档有助于找到更相关的真实内容。


> **（原图转文字）HyDE 实现逻辑与流程**
>
> **逻辑**：基于 Query 生成假设性回答文档，用该「假文档」向量在真实库检索，缓解 Query 与原文之间的语义鸿沟。  
> **优势**：缓解冷启动检索问题；增强向量空间重合度。
>
> **流程**：用户 Query → LLM 生成 Hypothetical Document → 假设文档向量化 → 用假文档向量检索真实库 TopK → 拼接上下文 → 上下文+Query 生成最终答案。




### 16.Fusion

> **（原图转文字）Fusion：向量 + BM25 + RRF**
>
> **定义**：向量（语义）检索 + BM25（关键词）检索，经 RRF 融合。  
> **假设**：多路检索可互补。  
> **逻辑**：向量找「意思相近」，关键词找「精确命中」，再融合排序。  
> **优势**：几乎无漏检；鲁棒性强。
>
> **示例**（问「如何预防感冒？」）：语义可能召回免疫力相关/甚至新冠预防等近义但偏题内容；关键词更易命中「流行性感冒预防」；RRF 后把既语义相关又含准关键词的文档排到前面。




### 17.CRAG(Corrective RAG)

> **（原图转文字）CRAG（Corrective RAG）定义与逻辑**
>
> **定义**：C = Corrective（纠错）。评估检索相关性，不佳则转 Web 搜索。  
> **假设**：系统需能评估检索质量，并在必要时换信息源。  
> **逻辑**：对召回内容打高/中/低相关分——高：直接答；中：结合 Web 补充；低：主要靠 Web。


> **（原图转文字）CRAG 流程图**
>
> 用户 Query → 检索文档库（召回相关内容）→ **评估相关性（高/中/低）**  
> - **高相关**：直接生成答案  
> - **中相关**：结合 Web 搜索结果生成答案  
> - **低相关**：主要用 Web 搜索结果生成答案  
> → 最终答案输出




## 当前客服RAG-SOTA 

### step1: BM25+Dense+RRF -->Top 100(粗召回)

BM25是关键词检索 ：基于词频（TF）、逆文档频率（IDF）和文档长度，计算查询词与文档的关键词匹配度，擅长处理客服问题中的品牌词、术语、政策编号等精确关键词，解决 “同义词不匹配” 的问题，比如用户搜 “退款政策”，能匹配到包含 “退货退款” 的文档

Dense是稠密向量检索（语义检索）：用 Embedding 模型将用户问题和文档片段都转为向量，通过余弦相似度匹配语义相近的内容，擅长处理口语化、模糊的客服问题，解决 “关键词不匹配但语义相关” 的问题，比如用户说 “我想退掉刚买的东西”，能匹配到 “退款流程” 的文档

RRF是倒数排名融合：对 BM25 和 Dense 各自返回的排序结果，按排名倒数加权融合，得到统一的综合排序，避免单一检索的偏差，比如 BM25 漏召回的语义相关文档，或 Dense 召回的语义相近但关键词不匹配的文档，都能被保留

### Step2：cross-encoder rerank -->Top 5-10

这一步是**精排阶段**，目标是对粗召回的 Top 100 候选片段，进行更精准的相关性打分，筛选出最有价值的 Top 5-10 个片段，作为最终给大模型的上下文。

粗召回阶段的 Dense 检索用的是**双编码器（bi-encoder）**：问题和文档分别编码，无法捕捉两者的细粒度交互，只能计算整体相似度，容易出现 “语义相近但实际无关” 的误召回；

cross-encoder 则是将「问题 + 文档片段」拼接成一个输入，让模型直接输出两者的相关性分数，能捕捉到更细粒度的语义匹配，比如用户问 “7 天无理由退款”，能精准区分 “商品支持 7 天无理由” 和 “活动截止 7 天” 的差异。



### Step 3: LLM with citations

这一步是**生成阶段**，目标是让大模型基于重排后的 Top 5-10 片段，生成准确、合规的客服回答，并标注信息来源。

#### 1. 带引用（citations）的核心价值

- **客服场景的合规性**：客服回答必须基于官方知识库，不能编造信息，引用能直接对应到具体的文档片段，方便溯源和审计；
- **提升用户信任度**：用户可以看到回答的信息来源，避免被错误信息误导；
- **降低幻觉风险**：强制模型基于提供的上下文生成回答，减少大模型的 “一本正经胡说八道”。

#### 2. 常见实现方式

- 让模型在生成回答时，用特定格式（如 `[1]` `[2]`）标注引用的片段编号；
- 后端可以根据编号，直接关联到原始文档的链接或段落，方便用户查看原文。



### Anthropic Contextual Retrieval 优化技巧

这是图中额外补充的一个**召回增强方案**，也是当前很多顶尖 RAG 系统采用的优化手段，核心解决「chunk 上下文缺失导致的 Embedding 匹配偏差」问题。

### 1. 问题背景

传统 RAG 会把长文档切成固定长度的 chunk（比如 512 tokens），但 chunk 本身可能脱离了原始文档的上下文，比如：

- 原文档是 “【退款政策】用户在签收后 7 天内，可申请无理由退款，商品需保持完好”，切分后 chunk 变成 “用户在签收后 7 天内，可申请无理由退款，商品需保持完好”；
- 当用户问 “无理由退款的条件是什么？” 时，chunk 里没有 “退款政策” 这个关键词，Embedding 模型可能无法精准匹配。

### 2. 优化方案

**在对 chunk 做 Embedding 之前，先用 Anthropic 的 Haiku 轻量大模型，为每个 chunk 生成一段「上下文描述」，并拼接到 chunk 前面，再一起做 Embedding。**

比如上面的 chunk，会被增强为：

> “这段文字是关于退款政策的，说明用户在签收后 7 天内，可申请无理由退款，商品需保持完好。用户在签收后 7 天内，可申请无理由退款，商品需保持完好”

### 3. 核心收益

- 为 chunk 补充了文档层面的上下文信息，让 Embedding 向量能更好地捕捉 chunk 的语义主题；
- 显著提升了粗召回阶段的语义匹配准确率，尤其是对于碎片化、脱离上下文的 chunk；
- Haiku 模型成本低、速度快，适合批量处理海量 chunk，不会带来过高的额外开销。

### 整体方案的优势与适用场景

### 1. 优势

- **召回全面**：BM25 + Dense + RRF 混合检索，兼顾关键词和语义，召回率远高于单一检索；
- **排序精准**：cross-encoder 重排大幅提升了候选片段的相关性，减少了给模型的噪音信息；
- **回答合规**：带引用的生成，解决了客服场景的幻觉和溯源问题；
- **工程友好**：各阶段的计算成本逐步降低（粗召回高效、精排和生成只处理少量数据），能支撑高并发的客服场景。

### 2. 适用场景

这个方案是当前客服、知识库问答场景下的通用 SOTA 架构，尤其适合：

- 知识库文档量大、用户问题口语化 / 模糊化的场景；

- 对回答准确性、合规性、可溯源性要求高的场景（如电商客服、金融客服、政务咨询）。

  

## RAGAS 打分机制

传统指标（BLEU(n-gram),ROUGE(关键词)）只比较字面重叠，忽视语义是否相同。完全无视检索上下文（是否利用了上下文，上下文利用率）。无法检测幻觉（最核心的风险）

#### Faithfulness（忠实度）：输入 问题+检索文档+生成的答案

LLM把答案拆成独立声明 --> 逐条对比检索文档：是否被检索上下文支持 --> 得分=被支持声明数/总声明数  (0~1)

局限性：不管文档检索的对不对，只要是文档记载的就算支持（只测忠实度，不测真实性）

#### Answer Relevance（答案相关性）：输入 原始问题+生成的答案（无需文档）

LLM从答案反向生成N个问题 --> 算反向问题于原问题的余弦相似度 --> 得分 = 平均余弦相似度

#### Context Relevance（上下文相关性）：输入原始问题+检索文档（不看答案）

挑出文档中对回答有用的句子 --> 得分=有用句数/文档总句数

#### 局限性（LLM给LLM评分）

位置偏差：答案换顺序，相当比例的判断可能会反转

冗长偏差：长答案更容易被偏好

自我偏好：自己评自己得分会更高

家族偏好：同一系列的LLM彼此之间打分会偏高

缓解方案：

（工程手段）答案换顺序取平均/换模型家族/多裁判投票取平均/简洁性维度缓解冗长偏差

（微调小模型）专用小模型当裁判

（人及混合）大部分样本自动评，高风险和边界case由人工评

## 工业级RAG向量数据库全链路构建

### 多源数据接入与智能解析

#### 基础数据流

多格式（PDF,Word，HTML，图片等）抽取，剔除乱码，页眉页脚，保留文档层级结构

#### 🧩 第一步：多格式文档解析

这是基础，目标是将不同格式的文件转化为统一的、可处理的文本或结构化数据。

- **策略：模块化解析器**。为每种文件类型配备专用解析器。
- **PDF**：
  - **文本型PDF**：使用`pdfplumber`、`PyPDF2`等库提取文字。
  - **扫描件/图片型PDF**：必须使用OCR技术识别文字，常用的有`PaddleOCR`、`Tesseract`。
- **Word/PPT**：使用`python-docx`、`python-pptx`等专用库。
- **HTML**：使用`BeautifulSoup`等库解析标签，提取正文。
- **图片**：直接使用OCR引擎进行文字识别。
- **一体化工具**：一些工具直接提供了多格式支持，例如**MinerU**、**Docling**等。

#### 🧹 第二步：文档清洗

清洗的目的是去除噪声，保留精华，为后续处理提供高质量的数据。

#### 1. 剔除乱码

- **编码统一**：将所有文本统一转换为标准编码，如 **UTF-8**。
- **修复断裂**：修复因PDF解析导致的换行断裂，如将“机 器 学 习”还原为“机器学习”。
- **过滤字符**：使用正则表达式等方法过滤掉控制字符、特殊符号等无效字符。

#### 2. 去除页眉页脚

这是提升检索质量的关键一步，否则检索时可能匹配到大量噪声。

- **基于规则的过滤**：
  - **位置信息**：利用文档的布局信息，识别并剔除位于页面顶部或底部的重复文本。
  - **正则表达式**：匹配如“版权所有”、“第X页”等固定格式的文本。
- **基于AI的识别**：
  - **布局分析**：使用`LayoutLM`、`LayoutParser`等模型，通过分析视觉结构（如字体、大小、位置）来智能识别页眉页脚区域。
  - **语义理解**：利用LLM对文本片段进行分类，判断其是否为页眉页脚等无关内容。

#### 3. 其他清洗操作

- **去重**：删除重复的章节标题、段落等内容。
- **格式规范化**：合并多余空格、空行，使文本更规整。

#### 🏗️ 第三步：保留文档层级结构

这是高级功能，能让RAG系统理解文档的“骨架”，实现更精准的检索。

- **策略一：利用现有结构**。如果PDF文档本身带有**书签（Bookmarks）或大纲（Outlines）**，可以直接提取，构建出精确的层级树，这是最快且零误差的方法。
- **策略二：基于规则的提取**。
  - **标题检测**：通过分析**字体大小、样式（如加粗）** 来识别标题及其层级。
  - **正则匹配**：用正则表达式匹配“第X章”、“1.1”、“1.1.1”等中英文章节标题模式。
- **策略三：基于AI的布局分析**。使用**版面分析（Layout Analysis）** 模型，将文档页面划分为标题、段落、表格、图片等不同区域，并识别其阅读顺序。
- **输出格式**：解析完成后，通常将带有层级结构的信息输出为**Markdown**或**JSON**格式，以便后续处理。



### 分块以及策略

#### 一、模块总述

文本分块是解析后、向量化入库前核心步骤，目的解决 Embedding 模型输入 Token 上限、大模型上下文窗口限制；普通硬切分会割裂语义、出现代词指代丢失，分**基础通用方案**与**专家级优化方案**两套体系，不同业务文档匹配对应策略。

#### 二、基础分块规则（通用兜底方案）

#### 1. 两种基础切分实现

1. Token 切分（固定长度滑动切分）

   按 Embedding 模型原生 Token 计数器截断文本，设置固定 Chunk 长度 + 重叠 Overlap。

2. 递归字符切分

   按层级分隔符递归切割：标题

   ```
   #
   ```

   →段落换行

   ```
   \n
   ```

   →句号

   ```
   。
   ```

   →逗号

   ```
   ，
   ```

   ，优先在语义断点分割，减少句子被拦腰切断。

#### 2. 适用业务场景

| 分块方式       | 适用文档场景                                              | 不适用场景                                     |
| :------------- | :-------------------------------------------------------- | :--------------------------------------------- |
| Token 固定切分 | 纯短段落新闻、客服对话、无层级平铺说明文、短 FAQ          | 带多级标题、长篇技术手册、表格、多章节工业文档 |
| 递归字符切分   | Markdown 结构化文档、产品说明书、普通 Word 文档、课程讲义 | 无标点无分段的纯扫描转译大段文字、密集参数表格 |

#### 3. 基础方案痛点

仅靠分隔符 / 长度切割，长段落跨块时，下一块代词（它、该设备、此参数）丢失前文主语，检索后送入 LLM 易出现逻辑断裂、回答失真。

#### 三、专家级难点方案：解决语义截断、指代不明（工业级复杂文档专用）

针对**技术手册、工业标准、长篇合同、研发白皮书、多章节知识库**等长文本业务，三类优化手段组合使用。

#### 1. 滑动窗口重叠（10%~20% 重叠度）

#### 实现逻辑

每一段 Chunk 尾部保留 10%~20% 前文内容，下一段开头复用重叠文本，保证相邻切片语义衔接。

#### 适配场景

- 连续流程类文档：设备操作步骤、流水线工艺、故障排查流程；
- 线性叙事文本：政策条文、连载说明、连续实验记录；

#### 单独使用局限

仅缓解相邻块割裂，跨多块远距离指代、全局概念缺失问题无法解决。

### 2. 父子文档分块 Parent-Child Chunk（工业 RAG 首选）

#### 实现逻辑

1. **父块（大块）**：完整章节 / 完整段落，保留完整上下文语义，不入库向量库；
2. **子块（小块）**：从父块中切分短切片，存入向量数据库做相似度检索；
3. 检索流程：用户查询匹配到子块后，自动取出该子块归属的完整父块，全部送入大模型生成回答。

#### 适配业务场景

- 长篇工业技术手册、设备运维手册、多章节产品白皮书；
- 法律合同、标准规范、学术论文（单章节信息强关联）；
- 存在大量跨句 / 跨段代词、参数引用的专业文档；

#### 优势：从根源解决指代不明，检索上下文完整性最高。

### 3. 全局摘要注入

#### 实现逻辑

先对整篇完整文档生成全局核心摘要，**每一个子 Chunk 头部拼接该摘要**，让任意切片都自带文档整体背景信息。

#### 适配业务场景

- 多主题混合文档：合集类技术文档、多设备合集手册、多章节项目方案；
- 文档内存在大量全局统一术语、通用前提定义；
- 短句碎片化知识库、零散设备参数表；

#### 搭配建议：常与父子分块组合使用，小块检索时既有全局背景，又有完整章节上下文。

### 四、业务场景完整选型速查表（笔记核心）

| 业务文档类型                         | 推荐分块组合方案                           | 选择理由                                              |
| :----------------------------------- | :----------------------------------------- | :---------------------------------------------------- |
| 短 FAQ、客服对话、短资讯             | 基础：Token 切分 + 10% 滑动重叠            | 文本短、结构简单，轻量化足够，算力开销低              |
| Markdown 讲义、普通产品说明书        | 基础：递归字符切分 + 15% 重叠              | 自带标题段落层级，递归切分优先在语义断点分割          |
| 设备操作流程、连续工艺文档           | 递归字符切分 + 滑动窗口重叠                | 线性连续内容，重叠消除步骤断裂                        |
| 工业运维手册、技术标准、合同论文     | 父子文档分块 + 全局摘要注入 + 20% 滑动重叠 | 长章节强关联、大量代词 / 参数跨段引用，完整保留上下文 |
| 多设备合集、多方案混合文档、参数合集 | 递归切分 + 全局摘要注入                    | 文档主题分散，全局摘要补充统一背景定义                |
| 无分段纯扫描长文本、密集表格文档     | 父子文档分块（父块按页面划分）+ 全局摘要   | 无天然分隔符，依赖大块保存完整语义，摘要统一术语      |

### 五、笔记总结

1. 轻量化、短文本业务用**基础分块**即可满足需求，开发简单、推理速度快；
2. 工业级专业长文档、高精准知识库场景，必须叠加**专家级三套优化方案**，核心优先选用父子文档分块；
3. 三类专家方案各司其职：滑动窗口解决相邻块割裂、父子分块完整保留章节上下文、全局摘要补充文档全局背景，三者组合可彻底解决切片后语义截断、代词指代丢失的检索痛点。



### Embedding向量化

### 一、模块概述

本环节承接分块后的文本切片 Chunk，核心作用是将自然语言转换为计算机可计算的高维向量，解决文字无法做相似度检索的问题，分为**通用基础方案**与**垂直领域专有术语优化方案**，适配不同业务知识库。

### 二、基础特征提取（通用轻量化方案）

### 核心原理

调用开源 / 商用通用 Embedding 模型，把文本 Chunk 编码为**稠密向量（Dense Vector）**，向量维度通常数百～数千维，支持多语言文本统一编码。

### 两种基础能力

1. 稠密向量

   完整捕捉句子语义、上下文逻辑，依靠向量余弦相似度判断文本语义相近程度，是绝大多数 RAG 系统的基础检索载体。

2. 多语言支持

   内置中英日韩等多语种语义映射，混合语言文档无需拆分处理，适合通用资讯、跨境客服、通用百科知识库。

### 适用业务场景

| 方案                   | 适用场景                                         | 不适用场景                                   |
| :--------------------- | :----------------------------------------------- | :------------------------------------------- |
| 通用稠密向量 Embedding | 通用资讯、公开百科、通用客服 FAQ、多语言通用文档 | 医疗、法律、工业设备、企业内部专有术语知识库 |

### 基础方案核心痛点

通用模型仅基于公开通用语料训练，对行业专属缩写、内部黑话、专业名词语义理解偏差；检索时容易出现 “字面相近但语义无关”“专业术语匹配失效” 问题。

------

### 三、专家级难点方案：垂直领域 “黑话” 识别（行业知识库专用）

### 核心痛点

通用 Embedding 模型无法识别医疗、法律、工业、企业内部独有的专有名词、缩写、行业术语，导致专业文档检索召回率低、匹配错乱。

### 两套落地实现方法

#### 1. 领域微调 Embedding（对比学习微调）

##### 实现流程

1. 基于企业私有领域文档，批量生成领域专属问答对、同义术语样本；
2. 使用对比学习（Contrastive Learning）训练：同义术语 / 同类业务样本向量拉近，无关专业概念向量拉开；
3. 基于通用 Embedding 基座做增量微调，得到适配本行业的领域专属向量模型；

##### 适配场景

- 工业运维手册、医疗器械文档、法律合同、金融财报、企业内部技术规范；
- 存在大量自定义设备编号、内部流程代号、专业缩写的私有知识库。

##### 优势

从语义层面修正行业术语表征，长句、段落级专业文本检索精度大幅提升。

#### 2. 稀疏向量（Sparse Vector）混合检索

##### 实现逻辑

在稠密向量检索之外，新增稀疏向量维度（典型如 BM25、SPLADE），精准保留**关键词、专业术语、专有名词**的词级特征；

检索时采用稠密 + 稀疏混合打分（Hybrid Retrieval）：稠密向量负责整体语义，稀疏向量兜底精准术语匹配。

##### 适配场景

- 术语密集型文档：参数表、行业标准、药品名录、法条清单、设备零件清单；
- 缩写、代号高频出现的短文本知识库；

##### 优势

无需微调模型，低成本解决专业名词漏召回问题，可快速落地。

### 专家方案组合选型速查表

| 业务文档类型                             | 推荐向量方案组合                      | 选择理由                                       |
| :--------------------------------------- | :------------------------------------ | :--------------------------------------------- |
| 通用资讯、多语言 FAQ                     | 仅通用稠密 Embedding                  | 无行业术语，轻量化、算力成本低                 |
| 术语密集型参数 / 清单文档                | 通用稠密向量 + 稀疏向量混合检索       | 低成本，精准匹配专有名词、代号                 |
| 深度行业私有知识库（工业 / 医疗 / 法律） | 领域微调 Embedding + 稀疏向量混合检索 | 同时优化语义理解与术语精准匹配，工业级最优方案 |

------

### 四、笔记总结💡

1. 通用场景仅使用基础稠密 Embedding 即可满足需求，开发成本最低；

2. 垂直行业、含大量专业 “黑话” 的知识库，必须引入专家优化方案；

3. 短期快速落地优先选择**稠密 + 稀疏混合检索**；长期高精准需求，需基于私有领域数据微调专属 Embedding 模型；

4. 稀疏向量与领域微调分别从 “词级关键词”“句级语义” 两个维度解决专业术语识别缺陷，二者搭配可最大化检索召回准确率。

   

### 向量存储和索引构建

**通过“向量检索+结构化元数据”的双重过滤，并结合工程优化手段，来同时满足海量数据下的检索精度、召回速度和存储成本的要求**。

下面我来为你详细拆解其中的关键技术点与实现方法。

### 🧱 基础设计：向量、明文与元数据的“三合一”存储

基础库表设计存储了`向量 + 文本明文 + 元数据`三部分信息，这并非冗余设计，而是各有分工：

- **向量（Vector）**：由 `Embedding` 模型生成的高维向量，用于计算语义相似度，是检索的核心。
- **文本明文（Text Chunk）**：在检索后，将命中的向量对应的原始文本块返回给大模型，作为其回答的上下文依据。
- **元数据（Metadata）**：文档的“结构化标签”，如**作者、发布时间、文档类型（PDF/Word）、部门、保密等级**等。它不参与语义相似度计算，但能在检索前通过**预过滤（Pre-filtering）** 大幅缩小搜索范围，提升准确率。

### ⚙️ 索引构建：HNSW与余弦相似度的经典组合

- **距离度量**：使用**余弦相似度（Cosine Similarity）** 来衡量两个向量在语义空间中的方向一致性。这是目前语义检索最主流的度量方式，通常配合 `inner product` 实现。
- **ANN索引算法**：采用 **HNSW（分层可导航小世界图）**。它通过构建多层图结构，能将高维空间中的向量检索时间复杂度从暴力搜索的 `O(N)` 大幅降低到 `O(log N)`，是实现百亿级数据毫秒级召回的关键。

### 🚀 专家级难点方案：解决海量存储与数据更新难题

针对你提到的高成本与数据“脏”问题，图中给出了三套实用的工程解决方案：

1. **向量压缩以降低内存占用 (SQ/PQ)**
   - **问题**：HNSW索引需要将整个图的连接信息加载到内存，对于百亿级数据，内存开销可达TB级别，成本极高。
   - **方案**：采用**标量量化（SQ）** 或**乘积量化（PQ）** 压缩向量。
     - **SQ** 将每个向量的维度值从 `float32` 降为 `int8`，内存可减少约 **4倍**。
     - **PQ** 将高维向量切分为多个子向量并分别聚类，用聚类中心的ID（一个整数）来代表该子向量，可实现数十倍的压缩，但会牺牲部分精度。
   - **场景选择**：
     - **精度优先**（如医疗、法律）：使用SQ，损失小且恢复简单。
     - **成本优先**（如海量网页库）：使用PQ，牺牲少量精度换取极致的成本降低，甚至可以采用**PQ + HNSW**的混合方案。
2. **确保数据一致性 (软删除与覆盖更新)**
   - **问题**：源文档更新后，旧的向量索引若不删除，会导致检索结果包含“脏数据”。
   - **方案**：基于 **文档ID + MD5 Hash校验** 的软删除与覆盖更新（Upsert）机制。
     - **写入**：计算新文档内容的MD5值。如果MD5与旧值不同，则执行 **Upsert**（存在则更新，不存在则插入）。
     - **删除与清理**：采用**软删除（Soft Delete）**，即仅为文档添加一个 `is_deleted: true` 标记，而不立即从物理索引中移除。系统会定期（如每天凌晨）运行**后台Compaction任务**，批量清理这些标记为已删除的物理数据。这既能确保读操作的实时一致性，又能避免高频删除带来的性能抖动。
3. **缩小检索范围 (分区/分表)**
   - **问题**：在全球大客户场景下，每次检索都扫描全库索引，效率低下。
   - **方案**：根据元数据类别进行**分区（Partitioning）**。
   - **实践**：典型的做法是**租户隔离**。例如，为每个大客户（如“华为”）创建一个独立的 **Collection（集合）** 或 **Partition（分区）**。检索时，系统会根据请求中的 `tenant_id` 元数据，自动将查询路由到对应的分区，仅在该分区内执行HNSW检索。这种方式能将检索范围从千亿级缩小到千万级，极大地提升速度并降低资源消耗。

### 💎 总结

这张图所展示的方案，共同构成了一套从存储到索引的完整技术闭环：

- **存储层**：通过 `明文 + 元数据` 保证了可解释性与可过滤性。
- **索引层**：通过 `HNSW` 实现了高性能的语义近似搜索。
- **优化层**：通过 `SQ/PQ压缩`、`软删除+Compaction` 和 `分区/分表`，系统性地解决了工业级RAG在**存储成本**与**数据更新**两大核心痛点，最终实现了一个**高性能、低成本、可维护**的向量检索系统。



# RAG 减少检索冗余、重复内容的全套优化方案

## 一、检索阶段：从源头避免取出重复文档

### 1. 向量库去重入库（底层预处理）

1. 文档入库去重

   - 文本哈希：MD5/SHA256 计算全文 / 段落哈希，入库前比对，相同文本直接跳过；
   - 相似度过滤：新增片段与库内已有片段余弦相似度 > 0.95 判定重复，拒绝插入；
   - 分段粒度统一：固定 chunk 长度、重叠窗口标准化，避免同一段落切出多个高度相似块。

2. 元数据分区过滤

   

   给文档加来源、标题、章节、文件 ID 元数据，检索时限制

   同一文档最多返回 N 块

   （如单文件最多 2 段），防止单篇文档大量占满结果。

### 2. 检索结果后过滤（召回后剪枝）

1. 相似度聚类去重

   

   对 top-k 召回向量做 DBSCAN/K-means 聚类，每个簇只保留得分最高 1 条，丢弃同簇低分冗余片段；

2. 重复文本剔除

   

   两两计算召回文本相似度，超过阈值直接删除；轻量可用 SimHash 局部敏感哈希加速比对；

3. Top-N 分层截断

   

   不直接取 top100，先召回 top200，再内部去重后保留 top10–30 送入 LLM，减少冗余输入。

### 3. 多路检索融合去重（混合检索场景）

ES 关键词检索 + 向量检索会大量重复命中同一段落：

- 增加**文档 ID 去重**：合并两路结果，同一 doc_id 只保留最高分一条；
- 分数加权融合 + 去重排序，消除重复命中。

## 二、分块（Chunking）优化：从根源减少相似切片

冗余本质大多是切分不合理，同一信息被切成多段：

1. 语义分块替代固定长度分块

   

   用 LLM/TextTiling/Semantic Chunker 按语义边界切割，一个完整语义单元只生成 1 个 chunk，避免一句话拆多块；

2. 控制重叠窗口

   

   重叠不宜过大（建议 10%–15%），重叠越高重复内容越多；长文档优先低重叠；

3. 标题层级切块

   

   按文档标题、章节自动切块，每个章节独立 chunk，跨章节重复内容天然隔离。

## 三、重排（Rerank）阶段过滤冗余

1. 重排时附加重复惩罚

   

   BGE-Rerank、CrossEncoder 输入前先过滤高度相似文本，不送入重排节省算力；

2. 重排后二次去重

   

   重排完成按文档、段落相似度再次去重，保证送入 LLM 上下文无高度重复信息。

## 四、上下文压缩 / 摘要：消除内部冗余（送入 LLM 前）

### 1. 轻量文本压缩

- 去重段落合并：多段讲同一主题的片段合并、删减重复描述；
- SimHash 批量剔除高度重复短句。

### 2. 语义摘要聚合（高阶方案）

用小模型对召回的多条 chunk 做**主题聚合摘要**：

把多条相似文档浓缩为一段总结，完全消除重复，大幅减少上下文长度；

适合知识库大量重复政策、说明书类文档场景。

## 五、知识库增量更新防冗余

1. 增量更新时比对存量向量，只新增全新 / 新增信息片段，不重复入库已有内容；
2. 定期离线巡检向量库：批量扫描全库，删除重复、高度相似 chunk，释放存储同时减少未来检索冗余。

## 六、Prompt 工程抑制冗余输出（LLM 侧兜底）

即使检索带少量重复，通过提示词约束避免模型复述重复内容：

plaintext









```
要求：
1. 整合下面参考资料，自动合并重复信息，不要多次复述相同内容；
2. 同一知识点只说明一次，整合多份文档的互补信息，剔除冗余描述；
3. 回答简洁，不复制原文重复段落。
```

## 七、轻量化工程落地优先级（从易到难）

1. 最低成本：入库哈希去重 + 检索后 doc_id 去重 + 单文档返回数量限制
2. 中等成本：语义分块 + SimHash 召回去重
3. 高性能方案：向量聚类去重 + 多路检索融合去重
4. 高阶优化：语义聚合摘要 + 定期库内离线去重

## 补充：高性能去重工具选型

- 文本快速比对：SimHash、MinHash（百万级文本毫秒级）
- 向量聚类去重：FAISS DBSCAN、Milvus 聚类接口
- 语义分块：LangChain SemanticChunker、LlamaIndex SentenceSplitter 增强版
- 重排过滤：BGE-rerank + 自定义相似度过滤逻辑
