# 面试补强

## GIL方面

> **（原图转文字）GIL 三点提纲**
>
> 1. **核心机制**：CPython 全局解释器锁保证同一时刻仅一线程执行 Python 字节码，是内存管理保护机制而非设计缺陷。
> 2. **对不同任务的影响**：CPU 密集无法真正多核并行，多线程近似退化为单线程；IO 密集在等待时释放 GIL，多线程仍能提升并发。
> 3. **三种破局**：多进程（独立解释器）；C 扩展（计算下沉脱离 GIL）；asyncio（单线程事件循环处理高并发 IO）。


### GIL核心机制

GIL 全称 **Global Interpreter Lock，全局解释器锁**，仅存在于 CPython 官方解释器：

1. 核心规则：**同一时刻，整个进程内只能有 1 个线程执行 Python 字节码**；
2. 设计初衷：简化 CPython 内存垃圾回收、对象引用计数的线程安全管理，是早期开发的取舍，不是 bug；
3. 本质区分：它是**解释器层面的锁**，不是普通线程锁，只限制 Python 代码，不限制底层 C 代码。

#### 补充面试关键点

很多人误区：GIL 锁死多线程。实际是**线程会交替抢夺 GIL**：

- 计算任务：每执行固定字节码指令，主动释放 GIL 切换线程；
- IO 阻塞任务：触发网络 / 文件等待时，直接释放 GIL，让其他线程运行。

#### 二、GIL 对两类任务的性能影响（02）

#### 1. CPU 密集型（计算密集，如矩阵运算、循环大数计算）

- 限制：CPU 密集代码全程在运算、几乎没有 IO 阻塞，线程不会主动释放GIL，多核 CPU 无法并行，多线程只是线程来回切换，**速度不如单线程**；
- 原因：全程占用 CPU，几乎无 IO 等待，GIL 频繁切换产生额外开销，无法利用多核算力。

#### 2. IO 密集型（网络请求、文件读写、数据库查询）

- 优势：多线程能显著提速；
- 原因：线程发起 IO 操作后会阻塞等待，此时自动释放 GIL，其他线程可同时执行代码，充分利用等待空隙并发。
- 当程序发起 IO（读文件、发网络请求、查数据库）时，**CPU 根本没事可做，只能干等硬件完成操作**，这段空窗期就是 IO 阻塞。

#### 三、突破 GIL 限制的三种主流方案（03）

#### 方案 1：多进程 multiprocessing

- 原理：每个进程拥有**独立 CPython 解释器、独立 GIL、独立内存空间**，天然绕开 GIL；
- 适用：CPU 密集计算，充分利用多核；
- 缺点：进程创建销毁开销大，进程间通信（队列、管道）成本高。

#### 方案 2：C/C++ 扩展（Cython、NumPy、PyBind11）

- 原理：把重度计算逻辑下沉到无 GIL 约束的 C 层，计算时手动释放 GIL，多核并行运算；
- 典型案例：NumPy 矩阵运算、Pandas 底层、OpenCV，底层循环完全脱离 Python 字节码，不受 GIL 限制。

#### 方案 3：asyncio 异步 IO（单线程协程）

- 原理：单线程事件循环，用协程切换代替操作系统线程切换，IO 等待时自动切走其他协程；
- 适用：高并发 IO 场景（爬虫、Web 接口服务），比多线程切换开销更小；
- 局限：依然单线程，无法提升 CPU 计算速度，只优化 IO 等待并发。

#### 面试高频总结

1. GIL 是什么：CPython 全局锁，同一进程仅一线程跑 Python 字节码；
2. 什么时候多线程有用：IO 密集；什么时候没用：CPU 密集；
3. 多核计算用什么：多进程 / C 扩展；高并发 IO 用什么：asyncio 协程。



## 内存管理

> **（原图转文字）内存管理三驾马车**
>
> - **引用计数**：主力机制，引用归零立即回收。
> - **标记清除**：专门解决循环引用，遍历不可达对象清理。
> - **分代回收**：对象分 0/1/2 代，存活则晋升；老年代扫描频率降低，减少整体 GC 开销。


### 一、引用计数（基础核心机制）

#### 核心逻辑

每个 Python 对象内置`ob_refcnt`引用计数器，记录当前有多少变量 / 容器指向该对象：

1. 对象被赋值、传入函数、存入容器时，计数 + 1；
2. 变量销毁、重新赋值、函数执行结束时，计数 - 1；
3. 引用计数归零，内存**立即释放**，实时回收无延迟。

#### 优点

回收实时、逻辑简单，无全局扫描停顿，性能开销分散。

#### 致命缺陷

无法处理**循环引用**（如`a=[1];b=[2];a.append(b);b.append(a)`），两个对象互相引用，计数永远无法归 0，会造成内存泄漏。

------

### 二、标记清除（补充机制，解决循环引用）

专门弥补引用计数的短板，只针对**容器对象**（列表、字典、类实例等可存储其他对象的类型）。

#### 执行流程

1. **标记阶段**：从程序根对象出发遍历所有可达对象，标记存活对象；

2. **清除阶段**：未被标记的不可达对象判定为垃圾，统一回收释放内存。

   外部没有任何根引用指向两个互相循环引用的对象，这两个对象就可以清除了

#### 缺点

需要全局扫描对象，会产生 STW（程序短暂停顿），频繁全量扫描开销很高，因此引入分代回收做优化。

------

### 三、分代回收（性能优化策略）

#### 理论依据：弱代假说

新生对象大多短期失效，长期存活的老对象很少会变成垃圾。

#### 三代划分规则

| 代际           | 对象来源                 | GC 扫描频率      |
| :------------- | :----------------------- | :--------------- |
| 0 代           | 所有新创建对象           | 最高，回收最频繁 |
| 1 代           | 0 代 GC 后存活的对象     | 中等频率         |
| 2 代（老年代） | 1 代 GC 后存活的长期对象 | 极低，很少扫描   |

### 工作流程

1. 0 代对象数量达到阈值时，触发 0 代 GC，存活对象晋升 1 代；
2. 1 代累积到阈值，触发 1 代 GC，存活对象晋升 2 代；
3. 仅当 2 代达到阈值时，才执行完整全量标记清除扫描。

### 优化价值

大幅减少老年代扫描次数，降低全局 GC 停顿开销，显著提升整体内存回收效率。

------

### 三者协同关系总结

1. **引用计数**：日常主力回收，实时释放无引用对象；

2. **标记清除**：兜底方案，定期扫描清理循环引用垃圾；

3. 分代回收：优化标记清除性能，减少全量扫描开销。

   三者配合构成 Python 完整自动内存管理体系。

### 面试高频拓展考点

⚠️ 循环引用对象内存在 GC 时会被处理，但如果对象定义了`__del__`析构函数，可能出现对象无法被回收、内存泄漏的极端场景。

💡 可通过`gc`模块手动控制 GC 开关、查看分代阈值、强制触发垃圾回收。



## 核心数据结构底层实现

> **（原图转文字）核心数据结构底层实现提纲**
>
> - **List**：C 指针数组 + Over-allocation，扩容超额预分配，均摊追加 O(1)。
> - **Dict（3.6+）**：紧凑哈希，键值对连续数组 + 索引表，省内存且保留插入顺序。
> - **Set**：基于 Dict，元素作 Key、Value 为 None，O(1) 去重与成员判断。
> - **版本差异**：3.6 前 Dict 不保证顺序且内存开销大；3.6+ 紧凑布局成为标准。


### 一、List 动态数组

### 底层原理

1. 存储载体：C 语言结构体指针数组，数组内每个元素为 PyObject * 对象指针，统一指向 Python 对象。
2. 扩容策略：Over-allocation 超额预分配机制，实际存储 n 个元素时，会额外预留空闲槽位，减少频繁扩容带来的内存拷贝开销。
3. 时间复杂度：尾部追加、索引访问均摊 O (1)；中间插入 / 删除需要移动后续元素，时间复杂度 O (n)。

### 核心特性

- 支持随机索引访问；
- 有序、允许元素重复；
- 扩容时重新申请更大连续内存，拷贝原有数据后释放旧内存。

### 二、Dict 紧凑哈希表（CPython 3.6+）

### 3.6 版本新紧凑布局设计

1. 数据存储：键值对实体统一存放在**连续数组**，不再分散存储，大幅降低内存碎片、节省内存占用。
2. 索引分离：单独维护一张哈希索引表，索引表存储实体数组下标，通过哈希值映射找到对应条目。
3. 有序特性：键值对写入顺序与实体数组存储顺序完全一致，天然保留插入顺序。

### 版本演进差异

- Python 3.5 及更早：哈希表与键值对混合存储，内存开销大，**不保证遍历顺序**；
- Python 3.6：紧凑布局成为默认实现，内存占用减少约 20%~30%，插入顺序可稳定保留；
- Python 3.7：语言标准明确规定字典必须保留插入顺序，该特性正式标准化。

### 性能特点

键查找、新增、删除均摊 O (1)，极端哈希冲突场景退化至 O (n)。

------

### 三、Set 哈希去重集合

### 底层依赖

完全复用 Dict 哈希表逻辑：集合元素作为 Dict 的 Key，所有 Value 固定填充为`None`，仅利用哈希表的唯一性做去重。

### 核心能力

1. 成员判断 `x in set`：O (1) 时间复杂度，远快于 List 的 O (n) 遍历；
2. 自动去重：插入重复元素会直接覆盖，仅保留唯一值；
3. 无序（3.7 + 不保证顺序）、不可重复、元素必须可哈希（int/str/tuple 等）。

### 适用场景

数据去重、交集 / 并集 / 差集运算、快速存在性校验。

------

### 💡 面试拓展重点

1. List vs Set 查找对比

   百万级数据中判断元素是否存在：List 遍历全量 O (n)，Set 哈希查找 O (1)，性能差距可达上千倍。

2. Dict 内存优化亮点

   3.6 前老字典存在大量空哈希槽位，紧凑布局将键值对收拢为连续数组，是 Python 官方针对内存占用的关键优化。

3. 哈希冲突处理

   List/Dict/Set 均采用开放寻址法解决哈希碰撞，而非链表法；冲突时按固定偏移探测下一个空闲槽位。

4. 扩容触发条件

   Dict/Set 负载因子达到 2/3 时触发扩容，重新计算所有元素哈希并迁移至新数组。

### 四、三者核心对比速览

| 结构 | 底层基础       | 是否有序   | 可重复   | 查找复杂度 | 核心用途             |
| :--- | :------------- | :--------- | :------- | :--------- | :------------------- |
| List | 动态数组       | 有序       | 允许     | O(n)       | 有序存储、按索引取值 |
| Dict | 紧凑哈希表     | 3.6 + 有序 | Key 唯一 | O(1)       | 键值映射、结构化存储 |
| Set  | 复用 Dict 哈希 | 无序       | 禁止重复 | O(1)       | 去重、成员快速校验   |



> **（原图转文字）生成器与装饰器 / 深浅拷贝提纲**
>
> - **生成器**：`yield` 惰性求值，按需产出，适合大数据流与无限序列。
> - **装饰器**：基于闭包，不改原函数即可叠加横切逻辑（日志、鉴权、缓存）。
> - **浅拷贝**：只拷外层，内层共享引用。
> - **深拷贝**：递归拷贝全层级，适合需完全隔离的复杂结构。


## 生成器 & 装饰器

### 1. 生成器（Generator）

#### 核心原理

依靠 `yield` 关键字实现**惰性求值**，不会一次性把全量数据加载进内存，调用时才按需产出数据。

- 执行逻辑：函数运行到 `yield` 暂停，返回当前值；下次迭代从暂停处继续执行。
- 优势：内存占用极低，无需开辟完整数组存储结果。

#### 适用场景

1. 大数据流处理：百万级 SKU、日志逐行读取、MySQL 分批拉取数据；
2. 无限序列：自增 ID、实时数据流、持续消息消费；
3. 节省内存的遍历场景，替代一次性列表推导。

### 2. 装饰器（Decorator）

#### 底层基础

基于闭包（Closure）（记住外层函数局部变量的内层函数）实现，在不修改原函数源码、不改动调用方式的前提下，横向新增通用逻辑。

#### 典型业务落地场景（横切关注点）

- 日志记录：统一打印入参、出参、函数耗时；
- 权限鉴权：接口统一校验 Token、用户角色；
- 缓存加速：结果缓存（Redis），重复请求直接返回缓存；
- 异常捕获、接口限流、参数校验。

#### 特性：可叠加、无侵入，完美解耦业务代码与通用工具逻辑。

## 浅拷贝 & 深拷贝

### 1. 浅拷贝（`copy.copy()`）

- 行为：仅创建**外层容器新对象**，内部嵌套的子对象仍然和原对象共享同一份内存引用；
- 副作用：修改嵌套子对象时，原始对象的数据会同步被改动；
- 适用：仅单层简单数据（纯 int/str，无嵌套 list/dict），拷贝速度快。

### 2. 深拷贝（`copy.deepcopy()`）

- 行为：**递归遍历所有层级**，每一层对象都创建全新独立副本，完全切断与原对象的引用关联；
- 特性：修改任意层级数据，不会影响原始对象；
- 适用：多层嵌套复杂结构（商品多层属性、图谱嵌套数据、多维列表），需要完全数据隔离的场景。

### 关键对比

| 类型   | 拷贝层级     | 子对象引用   | 内存开销   | 适用场景         |
| :----- | :----------- | :----------- | :--------- | :--------------- |
| 浅拷贝 | 仅外层一层   | 共享原引用   | 小，速度快 | 单层简单数据     |
| 深拷贝 | 递归全部层级 | 全新独立对象 | 大，速度慢 | 多层嵌套复杂数据 |

------

### 💡 面试拓展补充

1. **生成器与列表推导区别**

```
# 列表推导：一次性加载全部数据，大数据易OOM
data = [i for i in range(1000000)]
# 生成器表达式：惰性生成，几乎不占内存
gen = (i for i in range(1000000))
```

1. 装饰器语法糖

   ```
   @decorator
   ```

    等价于 

   ```
   func = decorator(func)
   ```

   ，底层替换原函数引用；

2. 拷贝边界特例

   不可变基础类型（int/str/tuple）不存在拷贝风险，赋值即共享；只有列表、字典、自定义类等可变容器会产生引用问题。

## FastAPI 依赖注入（Depends）

### 1. 依赖注入核心作用

通过`Depends`将通用逻辑（数据库会话、用户鉴权、分页参数）抽取为可复用依赖函数，实现业务代码与基础设施解耦，提升代码可测试性、可维护性，是 FastAPI 核心特性

### 2. 自动解析与缓存机制

FastAPI 通过反射解析函数签名自动构建依赖链，支持嵌套依赖、同请求内缓存复用；避免重复创建数据库连接等资源，降低高并发场景系统开销。

### 1. 依赖注入解决了什么问题？

传统开发痛点：

- 鉴权、DB 连接、分页逻辑散落在每个接口，大量重复代码
- 业务逻辑与底层资源创建强耦合，单元测试需要大量模拟代码
- 每次请求重复初始化数据库会话、Redis 连接，资源开销高

依赖注入方案：

把公共逻辑抽成独立函数，用`Depends()`声明依赖，接口仅关注业务，资源由框架统一管理。

### 2. 解耦的具体体现

1. 复用公共逻辑

   数据库会话、Token 鉴权、分页、限流只写一次，所有接口直接引入

```
from fastapi import Depends, FastAPI
from sqlalchemy.orm import Session

# 通用依赖：获取DB会话
def get_db():
    db = Session()
    try:
        yield db
    finally:
        db.close()

app = FastAPI()
# 接口直接声明依赖，无需手动创建/关闭连接
@app.get("/user")
def get_user(db: Session = Depends(get_db)):
    return db.query(User).all()
```

1. 业务与基础设施分离

   接口只写查询逻辑，数据库连接、释放交给依赖函数管理；切换数据库实现时，仅修改get_db，不用改动所有接口。

2. 单元测试更简单

   测试时可直接传入模拟 DB、模拟登录用户，无需启动完整服务。

### 3. 自动解析与缓存机制详解

#### （1）自动构建依赖链（支持嵌套依赖）

框架通过反射读取接口函数、依赖函数的参数签名，自动递归解析多层依赖：

示例：鉴权依赖内部又依赖数据库会话

```
# 底层依赖
def get_db(): ...
# 嵌套依赖：依赖DB
def get_current_user(db: Session = Depends(get_db)):
    # 校验token、查询用户
    return user

# 接口直接使用上层依赖，框架自动递归加载get_db
@app.get("/profile")
def profile(user = Depends(get_current_user)):
    return user
```

执行链路：`get_db` → `get_current_user` → 接口函数，无需手动调用。

#### （2）单请求缓存复用（关键性能优化）

**同一个 HTTP 请求生命周期内，同一个依赖函数只会执行一次**，结果全局缓存复用：

- 若接口多个参数都依赖`get_db`，不会创建多个数据库会话
- 嵌套多层依赖共用同一个 DB 连接，避免频繁创建销毁连接
- 请求结束后，统一执行依赖内`finally/生成器收尾逻辑`释放资源

高并发收益：大幅减少数据库连接、网络客户端的创建次数，降低 CPU / 连接池开销。

------

### 三、对比 Flask / Django 同类能力

| 框架    | 依赖注入能力 | 实现方式                                                     |
| :------ | :----------- | :----------------------------------------------------------- |
| FastAPI | 原生完整支持 | `Depends`内置，自动解析 + 请求缓存、嵌套依赖、生成器资源管理 |
| Flask   | 无原生 DI    | 需手动封装全局工具、装饰器，无请求级缓存，资源释放需自行处理 |
| Django  | 弱依赖注入   | 中间件 / 类视图混入实现公共逻辑，不支持参数级声明依赖，无自动依赖链 |

### 四、核心优势总结

✅ **复用**：鉴权、DB、分页等通用逻辑统一抽取，消除重复代码

✅ **解耦**：业务代码不关心资源创建与销毁，底层组件可独立替换

✅ **易测试**：依赖可直接 Mock，不用复杂环境搭建

✅ **高性能**：单请求缓存复用资源，减少连接创建开销

✅ **灵活嵌套**：多层依赖自动递归解析，代码分层清晰

✅ **自动化**：基于函数签名反射，无需手动组装依赖关系

### 五、典型业务使用场景

1. 数据库 / Redis/MQ 会话管理（自动创建、请求结束自动关闭）
2. JWT/Token 用户登录鉴权、权限校验
3. 统一分页、排序、过滤参数解析
4. 请求日志、接口限流、跨域校验统一拦截
5. 文件上传公共校验、全局参数预处理



## FastAPI 请求生命周期完整链路全解析

这张图完整拆解了基于 Uvicorn + ASGI 架构的 FastAPI 单次 HTTP 请求全流程，分为三大阶段，下面逐段拆解原理、关键点与底层逻辑。

### 01 连接接收与中间件处理（网络层、横切逻辑层）

### 流程原文

Uvicorn 接收 TCP 连接后构造 ASGI scope 对象，中间件依次处理 CORS、日志记录、Trace ID 注入和请求耗时统计等横切逻辑。

### 底层拆解

1. 网络入口 Uvicorn

   FastAPI 本身不监听端口，依靠 ASGI 服务器（Uvicorn/Gunicorn+Uvicorn）接收 TCP 客户端连接，读取 HTTP 报文。

2. ASGI scope 对象

   ASGI 标准规范的请求载体，存储全部原始请求信息：请求路径、请求头、客户端 IP、协议类型等，是整个请求生命周期的原始上下文。

3. 中间件链式执行

   所有中间件按注册顺序串行执行，属于全局前置拦截，统一处理所有接口的通用横切能力：

   - CORS 跨域校验、OPTIONS 预检请求处理
   - 全链路 TraceID 生成 / 透传（分布式追踪）
   - 请求开始计时、访问日志打印
   - 全局鉴权、限流、异常捕获

> 特性：中间件在路由匹配**之前**执行，无论接口是否存在都会经过这一层。

------

### 02 路由匹配与参数解析（路由层、参数校验层）

### 流程原文

根据 HTTP 方法和路径匹配 endpoint，提取 Path、Query、Header、Body 参数，Pydantic 自动完成类型校验和转换。

### 底层拆解

1. 路由匹配逻辑

   同时匹配「请求方法（GET/POST/PUT）+ URL 路径」，找到对应的接口函数（endpoint）；匹配失败直接返回 404。

2. 四类参数自动提取

   FastAPI 自动从 scope 里拆分四类参数，无需手动解析原始请求：

   - Path：路径占位符 `/user/{id}`
   - Query：URL 后缀 `?name=xxx`
   - Header：请求头 Token、User-Agent
   - Body：POST/PUT JSON 请求体

3. Pydantic 强类型校验（FastAPI 核心优势）

   根据函数参数注解自动实例化 Pydantic 模型：

   - 自动类型转换（字符串转 int、日期解析等）
   - 自动参数合法性校验（长度、范围、正则、必填）
   - 校验失败直接返回标准化 422 错误，附带详细报错信息

> 此阶段完成后，所有入参已经校验完毕，合法参数会传递给依赖与接口函数。

------

### 03 依赖执行与响应返回（业务执行层、响应输出层）

### 流程原文

解析 Depends 依赖图注入 DB Session，async 函数直接 await 执行，同步函数放入线程池避免阻塞事件循环，最终序列化响应返回客户端。

### 底层拆解

1. 递归解析 Depends 依赖注入链

   通过反射读取接口 / 依赖函数签名，自动构建依赖树，按依赖顺序执行：

   - 单请求内依赖缓存复用：同一个 DB 会话、鉴权用户只实例化一次
   - 支持嵌套依赖（鉴权依赖数据库会话）
   - 生成器依赖自动管理资源：`yield db` 后自动执行 finally 关闭连接

2. 同步 / 异步接口差异化调度（ASGI 异步核心）

   - `async def` 异步接口：直接在事件循环 `await` 执行，无线程切换，高性能；
   - 普通 `def` 同步接口：自动丢到内置线程池运行，**不会阻塞全局事件循环**，解决同步 IO 拖垮整个服务的问题（Flask/Django 无此机制）。

3. 响应序列化与返回

   业务函数执行完成后：

   1. 自动将返回值序列化 JSON（字典、Pydantic 模型、ORM 对象自动转换）
   2. 包装标准 HTTP 响应对象（状态码、响应头）
   3. 经由中间件后置逻辑（记录耗时、打印返回日志）
   4. 通过 Uvicorn TCP 连接回传给客户端

------

### 完整串联总流程

客户端 TCP 请求 → Uvicorn 接收连接 → 构建 ASGI scope → 中间件全局处理 → 路由匹配接口 → 自动提取 + Pydantic 校验所有参数 → 递归执行 Depends 依赖（注入 DB / 用户等资源） → 区分同步 / 异步调度执行业务代码 → 结果序列化生成响应 → 反向经过中间件后置逻辑 → TCP 返回数据给客户端

### 和 Flask/Django 关键差异点

1. **异步原生**：全程 ASGI 事件循环，同步函数自动线程池隔离，不会阻塞服务；
2. **参数自动化**：无需手动解析 request 对象，Pydantic 内置校验；
3. **原生 DI 依赖**：请求生命周期统一管理数据库、鉴权等资源，自动复用与释放；
4. **分层清晰**：横切逻辑（中间件）、路由参数、业务依赖三层完全解耦。







## 分布式锁

单机锁（Python threading.Lock/ Java synchronized）只作用于**单台机器内多线程**；

分布式场景多台服务器独立内存，本地锁完全失效，必须借助**第三方共享存储**做全局互斥，这就是分布式锁，核心目标：同一资源同一时间仅一台节点持有锁。

### 方案一：MySQL 数据库实现分布式锁

分两种实现：悲观锁、唯一索引乐观锁

### 1）悲观锁：SELECT ... FOR UPDATE（行锁）

#### 原理

InnoDB 事务机制，查询时手动加**行排他锁**，其他事务阻塞等待；事务提交 / 回滚自动释放锁。

示例：商品库存表 `stock`，商品 ID=100

```
-- 开启事务
BEGIN;
-- 锁定id=100这一行，其他事务无法读写该行
SELECT stock_num FROM stock WHERE id = 100 FOR UPDATE;

-- 执行业务：扣减库存
UPDATE stock SET stock_num = stock_num -1 WHERE id=100;

-- 提交事务，自动释放锁
COMMIT;
```

#### 执行流程

1. 节点 A 开启事务，查询加行锁；
2. 节点 B 同时执行同一条查询，会阻塞，直到 A 提交释放锁；
3. 保证同一时间只有一个节点修改库存。

#### 致命缺陷

1. 锁依赖事务，**业务代码超长阻塞事务**会长期占用数据库连接，拖垮库性能；
2. 无自动过期：节点宕机没提交事务，锁永久占用，死锁；
3. 高并发下数据库连接池耗尽，吞吐量极低。

### 2）乐观锁：唯一索引（插入法）

#### 原理

新建一张锁表 `dist_lock`，字段 `resource_key` 加唯一索引；

获取锁 = 插入一条资源记录，唯一索引冲突则代表锁被占用；

释放锁 = 删除这条记录。

```
-- 获取锁：resource_key为锁定资源标识（stock_100）
INSERT INTO dist_lock(resource_key, holder) VALUES('stock_100','server-01');
-- 插入成功=拿到锁；报唯一冲突=锁被占用

-- 业务执行完成，释放锁
DELETE FROM dist_lock WHERE resource_key='stock_100' AND holder='server-01';
```

#### 补充优化

新增过期时间字段，定时任务清理长期未释放的锁，解决宕机死锁。

#### 适用场景

低并发后台管理系统、内部工具，不想引入 Redis/ZK 中间件，追求开发简单。

------

### 方案二：Redis 分布式锁（互联网高并发首选）

### 核心基础：SET key value NX EX（原子命令，重中之重）

#### 关键字参数解释

- `NX`：Only Not Exist，key 不存在才设置，存在直接返回失败（互斥核心）
- `EX`：设置 key 自动过期时间，解决节点宕机不释放锁导致死锁

#### 正确加锁命令（单条原子操作，不能拆分为两步）

```
# 锁定资源stock_100，持有者唯一标识uuid，过期30秒
SET stock_100 uuid-xxxx NX EX 30
```

❌ 错误写法（非原子，并发会丢锁）：

```
GET stock_100
# 判断不存在再SET，两步中间会被其他线程抢占
SET stock_100 uuid
EXPIRE stock_100 30
```

### 释放锁：必须 Lua 脚本，不能直接 DEL

#### 踩坑点

如果直接 `DEL stock_100`：A 的锁过期后，B 拿到锁，此时 A 执行完业务删除 key，会**误删 B 的锁**。

#### 正确 Lua 脚本（原子校验持有者 + 删除）

```
-- 参数1：锁key，参数2：当前节点唯一标识uuid
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

逻辑：先判断锁是自己持有的，才允许删除，杜绝误删。

### 核心进阶：Redisson 看门狗（WatchDog）解决锁过期问题

#### 场景痛点

设置锁过期 30 秒，但业务执行需要 40 秒，锁提前过期，其他节点提前拿到锁，并发冲突。

#### WatchDog 原理

1. 加锁时不手动设置过期时间；

2. Redisson 后台启动异步定时线程（默认 10 秒执行一次）；

3. 检测当前线程仍持有锁，自动将锁过期时间重置回 30 秒（续命）；

4. 业务执行完毕主动释放锁，定时任务停止续命；

5. 节点宕机，无续命线程，锁 30 秒后自动释放，无死锁。

   额外能力：原生支持可重入锁（同一节点多次加锁不会阻塞）、公平锁、读写锁。

### Redis 锁短板（AP 模型，一致性弱）

Redis 主从集群异步同步数据：

1. 主节点加锁成功，还未同步到从节点，主节点宕机；
2. 从节点升级为主节点，锁 key 不存在；
3. 其他节点可以重新加锁，出现**两把锁同时存在**，超卖 / 并发错误。

> 解决办法：金融强一致性场景不使用 Redis 锁，改用 ZK；普通电商、秒杀业务可接受极小概率故障，性能优先选 Redis。

### 适用场景

秒杀、订单扣库存、分布式定时任务、高并发接口，追求十万级 QPS 吞吐。

------

### 方案三：ZooKeeper 分布式锁（金融强可靠方案）

### 底层核心特性

1. **临时节点**：客户端与 ZK 断开 / 宕机，节点自动删除，天然防死锁；
2. **有序节点**：创建节点自动带递增序号；
3. **Watcher 监听机制**：节点删除时，监听的客户端收到通知。

### 完整加锁流程（以锁目录 `/lock/stock_100` 为例）

1. 客户端 A、B、C 同时在目录下创建**临时有序节点**：`lock-xxx-000001`、`lock-xxx-000002`、`lock-xxx-000003`；
2. 获取目录下所有子节点，排序，**序号最小的节点获得锁**（A 拿到锁）；
3. 未拿到锁的 B，监听前一个节点`000001`的删除事件；C 监听`000002`；
4. A 执行业务，完成后删除自己的节点；
5. B 收到节点删除通知，重新查询目录，此时自己序号最小，获取锁；

### 优势（CP 模型，强一致无丢锁）

1. ZK 集群数据同步是半同步，主节点写入必须过半从节点确认才算成功；主从切换不会丢失锁；
2. 临时节点自动清理，宕机不会死锁；
3. 天然公平锁，请求按顺序排队，不会出现饥饿；

### 短板

1. 性能远低于 Redis：创建 / 删除节点、Watcher 监听存在网络交互，并发吞吐量低；
2. ZK 侧重协调，不适合高并发高频加解锁场景；

### 适用场景

支付、资金交易、金融核心系统，**绝对不允许并发数据错乱**，牺牲性能换 100% 可靠。

------

### 三大方案横向对比表

| 实现方案            | 一致性模型  | 性能            | 防死锁           | 核心缺陷             | 适用业务             |
| :------------------ | :---------- | :-------------- | :--------------- | :------------------- | :------------------- |
| MySQL 数据库        | 强一致      | 极低            | 需手动清理过期锁 | 数据库瓶颈、连接占用 | 低并发后台系统       |
| Redis (含 Redisson) | AP 最终一致 | 极高 (10w+ QPS) | EX 过期 + 看门狗 | 主从切换可能丢锁     | 高并发秒杀、库存扣减 |
| ZooKeeper           | CP 强一致   | 中等            | 临时节点自动释放 | 吞吐量低             | 金融、资金核心业务   |

### 补充高频面试区分点

1. 追求性能、能容忍极低概率锁失效 → Redis Redisson；
2. 追求绝对数据可靠、不允许超卖 / 资金差错 → ZooKeeper；
3. 无额外中间件、简单低并发场景 → MySQL 锁；
4. 锁过期业务未执行完：Redis 靠看门狗续命，ZK 无此问题（节点不删除就一直持有）；
5. 可重入：Redisson 原生支持，ZK、MySQL 需要自行封装逻辑。
